# CLAUDE.md

减少常见 LLM 编码错误的行为准则。根据需要与项目特定指令合并。

**权衡：** 这些准则偏向谨慎而非速度。对于简单任务，自行判断。

## 1. 编码之前先思考

**不要假设。不要隐藏困惑。呈现权衡。思考一步超越眼前的错误。**

在实现之前：
- 明确陈述你的假设。如果不确定，询问。
- 如果存在多种解读，呈现它们——不要沉默地选择一个。
- 如果存在更简单的方法，说出来。有必要时提出异议。
- 如果有不清楚的地方，停下来。说出困惑所在。询问。

在修复错误或警告时：
- 不要仅仅消除编译器告警——理解那些死代码或错误代码的本意，然后彻底移除它。
- 向外追踪一跳：如果移除 X 导致 Y 不再有读取者或调用者，Y 也要移除。在同一次提交中完成，不要留到后续修补。

## 2. 简洁优先

**解决问题所需的最小代码。不做任何投机。**

- 不添加未被要求的功能。
- 不为单次使用创建抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能的场景做错误处理。
- 如果你写了 200 行，而它可以压缩到 50 行，重写它。

问自己："一位资深工程师会认为这过于复杂吗？"如果是，简化它。

如果一个修复留下了没有 getter 的 setter、没有使用的 import、或永远不会执行的逻辑路径，它就是不完整的。移除一个变量意味着移除一切仅为服务它而存在的东西。

## 3. 外科手术式变更

**只触碰你必须碰的。只清理你自己的烂摊子。**

在编辑已有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有坏的东西。
- 匹配现有风格，即使你会有不同的选择。
- 如果你注意到无关的死代码，提及它——不要删除它。

当你的变更产生孤儿时：
- 移除你的变更使其不再使用的 import/变量/函数。
- 除非被要求，否则不要移除已存在的死代码。

检验：每个被修改的行都应该直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环直至验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后使其通过"
- "修复 bug" → "编写一个能复现它的测试，然后使其通过"
- "重构 X" → "确保重构前后的测试均通过"

对于多步骤任务，陈述简要计划：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

强有力的成功标准让你能独立循环。薄弱的成功标准（"让它能用"）需要不断澄清。

---

**这些准则正在生效的标志是：** diff 中更少的不必要变更、更少的因过度复杂而重写、以及澄清性问题在实现之前而非在错误之后被提出。
